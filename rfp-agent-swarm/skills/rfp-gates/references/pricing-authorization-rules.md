# Pricing Authorization Rules

Used by the Account Executive at Gate 3. Codifies the organisation's
non-negotiable rule that pricing is NEVER generated by AI, the authorisation
hierarchy for discounts, and the evidence every pricing response must carry.

---

## 1. Fundamental rule

Pricing is NEVER generated by the LLM. The Pricing Specialist (or rfp-respond
when operating in structured mode) STRUCTURES a pricing response from entries
in `authorised_pricing_inputs.json`. No other source of pricing is permitted in
the response package.

A pricing response that fails this rule is an auto-fail at the gate.

---

## 2. Discount tiers (placeholders — calibrate per org policy)

| Tier | Threshold | Authorising role |
|---|---|---|
| Standard | up to X% off list | Account Executive (self-serve) |
| Managerial | X% – Y% | Sales Manager |
| VP | Y% – Z% | VP Sales |
| Exec | above Z% | CRO + CFO jointly |

`X`, `Y`, `Z` are set in `working/pricing_policy.json` and reviewed quarterly.
The gate reads the current thresholds at runtime; the script does not hard-code
them.

---

## 3. Required fields per pricing response

| Field | Purpose | Auto-fail if missing? |
|---|---|---|
| `pricing_input_id` | Reference to an entry in `authorised_pricing_inputs.json` | Yes |
| `pricing_version` | Version of the input used | Yes |
| `source` | Must be `structured-from-authorised-input`. Never `generated`. | Yes |
| `discount_pct` | Applied discount (0 if list price) | Yes |
| `discount_tier` | Standard / Managerial / VP / Exec | Yes if discount > 0 |
| `authorisation_id` | ID of the authorisation record from the relevant role, required when tier > Standard | Yes when applicable |
| `approved_by` | Name of authoriser | Yes when applicable |
| `approved_on` | UTC timestamp | Yes when applicable |
| `currency` | ISO 4217 code | Yes |
| `validity_until` | Quote expiry | Yes |

---

## 4. Auto-fail conditions

| Condition | Code |
|---|---|
| Pricing response with `source=generated` | PRICING_GENERATED |
| Missing `pricing_input_id` | PRICING_UNLINKED |
| `pricing_input_id` not present in `authorised_pricing_inputs.json` | PRICING_INPUT_UNKNOWN |
| `pricing_version` does not match the current version on that input | PRICING_VERSION_STALE |
| Discount above Standard tier without `authorisation_id` | PRICING_DISCOUNT_UNAUTHORISED |
| Discount in an unauthorised currency | PRICING_CURRENCY_UNAUTHORISED |
| Quote validity in the past | PRICING_EXPIRED |
| Bundle assembled from individually authorised inputs but bundle itself not authorised | PRICING_BUNDLE_UNAUTHORISED |

---

## 5. Account Executive sign-off responsibilities

At the Pricing Gate, the AE confirms:

1. Every line item links to an authorised pricing input at the correct version.
2. All discounts are within tier OR have a valid authorisation ID.
3. The commercial package is consistent with the opportunity value and the
   deal's strategic importance.
4. Any MFN or price-hold commitments in the buyer's ask have been flagged with
   Legal (cross-gate dependency).
5. The currency, quote validity, and payment terms match the opportunity
   record in CRM.

The AE's approval captures acceptance of commercial risk on behalf of the
organisation. This is not a rubber-stamp of the automated precheck.

---

## 6. Cross-gate interactions

| Legal/Pricing interaction | Resolution |
|---|---|
| Buyer requests MFN pricing clause | Legal flags Blocker; AE escalates to CRO if strategic |
| Buyer requests multi-year price lock | AE confirms viability against forecast; Legal confirms clause language |
| Buyer requests bundled security feature at list | Pricing Ops confirms bundle authorisation; SecOps confirms product availability |

---

## 7. Evidence pack produced for the AE

The Pricing Gate precheck emits a table like:

| question_id | pricing_input_id | version | discount_pct | tier | auth_id | auth_by | issues |
|---|---|---|---|---|---|---|---|
| Q-PR-001 | PRI-SAAS-STD-12 | v3 | 0% | Standard | — | — | ok |
| Q-PR-002 | PRI-SAAS-ENT-36 | v2 | 18% | Managerial | AUTH-2026-00451 | j.manager@example.com | ok |
| Q-PR-003 | PRI-ADDON-SEC | v1 | 35% | Exec | — | — | PRICING_DISCOUNT_UNAUTHORISED |

Rows with issues block the gate request from being sent until rectified.

---

## 8. What the AE does NOT rubber-stamp

- Product feasibility claims that imply engineering effort the roadmap has not
  committed.
- Security posture statements that affect pricing (e.g. HYOK surcharge
  waiver).
- Legal clauses that have commercial consequences (e.g. service credit
  escalation).

These remain with the Security and Legal gates respectively.
