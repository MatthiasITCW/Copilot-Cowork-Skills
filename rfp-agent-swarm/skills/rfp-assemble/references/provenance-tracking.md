# Provenance Tracking

Provenance is the contract this skill holds with the wider RFP Agent Swarm:
every answer that appears in the deliverable must trace back to either a KB
entry or a logged `generated+reviewed` decision. Provenance is what makes
the system auditable and what makes the learning loop work.

---

## 1. Mandatory Provenance Fields

Every assembled response carries all of the following fields. Missing fields
cause `assemble_document.py` to fail the build.

| Field | Type | Description |
|:---|:---|:---|
| `response_id` | string | Unique identifier for the response (e.g. `R-00041`) |
| `question_id` | string | Matching question identifier from the RFP parse (e.g. `Q-00041`) |
| `category` | string | Security / Technical / Commercial / Company |
| `source` | string | `bank_entry:<KB_ID>` or `generated+reviewed` |
| `tier` | string | `HIGH` / `MEDIUM` / `LOW` |
| `reviewer` | string | Email of the human reviewer who signed off |
| `review_status` | string | `approved` / `approved_with_changes` |
| `last_updated` | ISO-8601 | Timestamp of the final approval |

Optional but recommended:

| Field | Type | Description |
|:---|:---|:---|
| `kb_confidence` | number | Raw reranker score from Azure AI Search |
| `correction_applied` | boolean | True if reviewer changed the draft |
| `generation_model` | string | Model name for LOW-tier only (e.g. `gpt-4o`) |

## 2. Embedding Rules by Format

Each deliverable format embeds provenance differently because the formats
have different capabilities.

| Format | Embedding |
|:---|:---|
| Word (.docx) | Dedicated provenance appendix section + optional Word comments on any `correction_applied: true` answer |
| Excel (.xlsx) | Extra columns in each row (hidden by default but retained for audit) |
| PDF (.pdf) | Dedicated provenance appendix section (flattened along with the rest of the PDF) |
| Portal CSV / JSON / XML | Side-car `manifest.json` packaged in the same zip |

### Word — appendix format

A table of at least 8 columns (the mandatory fields above), paginated, with a
cover heading "Provenance Audit". Word comments are added to cells for any
response where `correction_applied` is true so reviewers of the final
document can see what changed.

### Excel — extra columns

Columns F–H (and I if needed) carry provenance. By default these columns are
hidden so the buyer's view is clean, but they are not removed. An auditor
using the source file can unhide them.

### PDF — appendix

Same layout as the Word appendix. Because PDFs are flattened, this is
read-only.

### Portal formats — side-car manifest

Portal CSV / JSON / XML specifications rarely include free-form provenance
fields. To preserve provenance without breaking the portal import, a
`manifest.json` file is placed alongside the portal file inside the zip.
Structure:

```
{
  "rfp_id": "...",
  "provenance": [
    {
      "response_id": "R-00041",
      "question_id": "Q-00041",
      "category": "Security",
      "source": "bank_entry:KB-0925",
      "tier": "HIGH",
      "reviewer": "jane.doe@example.com",
      "review_status": "approved",
      "last_updated": "2026-04-20T14:22:03Z"
    }
  ]
}
```

## 3. Audit Requirement

The provenance appendix row count **must equal** the total question count.
This is a hard invariant:

| Rule | Enforcement |
|:---|:---|
| Total questions == total provenance rows | `assemble_document.py` exits non-zero if mismatch |
| Every response has all mandatory fields | Validated during manifest build |
| No duplicate `response_id` | Validated during manifest build |
| `source` is either `bank_entry:<ID>` or `generated+reviewed` — no other values | Validated |
| `review_status` is `approved` or `approved_with_changes` only | Validated |

A mismatch means either a response was dropped or a duplicate was created.
Both are correctness failures — not to be silently auto-repaired. The skill
stops and surfaces the mismatch for the human to resolve.

## 4. Generated + Reviewed — Why This Class Exists

Responses with no KB match are generated by the responding agents. These can
only reach the deliverable if a human reviewer in `rfp-review` explicitly
approves the generated text. Their provenance marker is
`generated+reviewed` — not simply `generated`, because pure generation is
never allowed to ship.

The distinction matters for two reasons:

1. **Audit** — leadership needs to know how much of the deliverable is
   KB-traceable vs human-approved generation.
2. **Learning loop** — `rfp-answer-bank` consumes the `generated+reviewed`
   set as candidates for new KB entries. A high count here signals a gap in
   the KB that should be filled.

## 5. Retention

| Artefact | Store |
|:---|:---|
| Final deliverable + cover letter + analytics | Dataverse (SharePoint-backed) |
| Provenance record (JSON) | Dataverse — alongside the deliverable |
| Corrections log | Dataverse — feeds `rfp-answer-bank` |
| Gate audit log | Dataverse |
| Raw reviewed responses | Dataverse (retention per org policy) |

Retention period for provenance records matches the RFP record retention
policy of the organisation — typically 7 years for customer-facing
submissions.

---

## 6. Read-Back Patterns

Common questions a human or downstream agent asks against the provenance
record:

| Question | Query |
|:---|:---|
| "What fraction of this RFP was KB-matched?" | `count(source startswith 'bank_entry:') / total` |
| "Which responses did Jane Doe review?" | `reviewer == 'jane.doe@example.com'` |
| "Show me every `generated+reviewed` answer" | `source == 'generated+reviewed'` |
| "Which KB entries were most reused on this RFP?" | group by `source`, count |
| "Were any answers updated after submission?" | `last_updated > submission_date` |

These queries run against the Dataverse-stored JSON — not against the
deliverable file itself. The deliverable is the shipped artefact; the
JSON is the source of truth for auditors.
